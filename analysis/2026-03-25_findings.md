## Findings

### 1. Critical - Security - CSRF exposure on cookie-authenticated write endpoints

The current auth model is vulnerable to CSRF.

- The session cookie is issued with `SameSite=None`, which is required for cross-origin cookie use:
  - [apps/api/src/modules/auth/auth.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/auth/auth.routes.ts#L61)
  - [.env.example](C:/Users/song/Desktop/pcu_graduationproject_v2/.env.example#L12)
  - [README.md](C:/Users/song/Desktop/pcu_graduationproject_v2/README.md#L83)
- State-changing endpoints rely on the session cookie and do not add CSRF protection such as token verification or strict `Origin` / `Referer` checks:
  - [apps/api/src/plugins/auth.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/plugins/auth.ts#L50)
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L182)
  - [apps/api/src/modules/assets/assets.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/assets/assets.routes.ts#L79)

Impact:

- A malicious external site could trigger project submission, asset upload, member modification, or asset deletion on behalf of a logged-in user.
- `multipart/form-data` endpoints are especially relevant here because they are often easier to trigger cross-site than custom JSON requests.

Recommendation:

- Add CSRF protection on all mutating endpoints.
- At minimum, validate `Origin` against the allowed frontend domain.
- Prefer a dedicated CSRF token for cookie-authenticated writes.

### 2. High - Security / Wiring - Regular `USER` can publish directly

The current permission model allows normal users to publish projects directly.

- Submission accepts `autoPublish`:
  - [apps/api/src/shared/validation.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/shared/validation.ts#L44)
- The backend writes `PUBLISHED` when `autoPublish` is true without role gating:
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L218)
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L260)
- The update API also accepts direct status changes:
  - [apps/api/src/shared/validation.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/shared/validation.ts#L26)
- Access control only ensures ownership and draft-only editing for non-privileged users:
  - [apps/api/src/modules/admin/project-access.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/project-access.ts#L24)
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L137)
- The frontend explicitly exposes the immediate publish option to users:
  - [apps/web/src/pages/admin/AdminProjectNewPage.tsx](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/web/src/pages/admin/AdminProjectNewPage.tsx#L139)

Impact:

- If the intended policy is "student submits, operator/admin approves", the current implementation violates that workflow.

Recommendation:

- Enforce publish/archive status transitions server-side by role.
- Ignore or reject `autoPublish` for `USER`.
- Keep moderation logic on the backend, not just in UI.

### 3. High - Wiring / Operations - `isUploadEnabled` exists but is not enforced

The year-level upload lock is present in schema and admin UI, but not in the actual submission flow.

- DB field exists:
  - [apps/api/prisma/schema.prisma](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/prisma/schema.prisma#L69)
- Admin endpoints and UI manage it:
  - [apps/api/src/modules/admin/admin-year.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-year.routes.ts#L15)
  - [apps/web/src/pages/admin/AdminYearsPage.tsx](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/web/src/pages/admin/AdminYearsPage.tsx#L88)
- But submission does not check it and even auto-creates missing years:
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L228)

Impact:

- The admin screen implies uploads can be locked per year, but students can still submit.
- Missing years can be created implicitly by user submissions, bypassing admin intent.

Recommendation:

- Validate `year.isUploadEnabled` during submit and asset-add flows.
- Remove implicit year creation for regular users, or reserve it for operators/admins.

### 4. Medium - Wiring - Protected download policies are enforced in backend but not fully exposed in UI

The backend correctly protects `SCHOOL_ONLY` and `ADMIN_ONLY` downloads, but the frontend does not provide a usable download action for those cases.

- Backend enforcement:
  - [apps/api/src/modules/assets/assets.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/assets/assets.routes.ts#L49)
- Public detail API returns the protected download URL:
  - [apps/api/src/modules/public/public.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/public/public.routes.ts#L128)
- The frontend only renders a clickable link for `PUBLIC`:
  - [apps/web/src/pages/ProjectDetailPage.tsx](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/web/src/pages/ProjectDetailPage.tsx#L110)

Impact:

- For `SCHOOL_ONLY` or `ADMIN_ONLY`, the system is technically capable but user-facing behavior is incomplete.
- Users only see guidance text, not an authenticated action path.

Recommendation:

- Show a download button for authenticated allowed users.
- Use `useMe()` to branch UI by auth and role.

### 5. Medium - Security / Wiring - Poster invariant is not enforced on the server

The server accepts any project asset as `posterAssetId`, without checking asset kind or readiness.

- Poster assignment endpoint only checks ownership within the same project:
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L395)
- Serialization assumes poster assets are public image-style assets:
  - [apps/api/src/modules/admin/admin-project.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/admin/admin-project.routes.ts#L60)
  - [apps/api/src/modules/public/public.routes.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/modules/public/public.routes.ts#L127)

Impact:

- A crafted request could set a `GAME` asset as poster.
- The DB would accept it, but the frontend would receive a public image URL that may 404 or behave incorrectly.

Recommendation:

- Restrict poster assignment to `IMAGE` / `POSTER` assets with `READY` status.
- Consider encoding that invariant more strongly in application logic or schema design.

### 6. Medium - Operations - Runtime env loading and docs are inconsistent

The runtime expects process environment variables directly, but the local setup guidance is misleading.

- Server validates `process.env` directly:
  - [apps/api/src/server.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/server.ts#L1)
  - [apps/api/src/config/env.ts](C:/Users/song/Desktop/pcu_graduationproject_v2/apps/api/src/config/env.ts#L45)
- README tells developers to copy `.env.example` inside `apps/api`, but the example file is in the repo root:
  - [README.md](C:/Users/song/Desktop/pcu_graduationproject_v2/README.md#L114)
  - [.env.example](C:/Users/song/Desktop/pcu_graduationproject_v2/.env.example#L1)

Impact:

- Local onboarding is likely to fail or confuse contributors.
- This is less about runtime correctness in production and more about setup reliability.

Recommendation:

- Either load env files explicitly in the API app, or correct the documentation so setup steps match reality.
- Keep env conventions consistent across root, API, and web.

### 7. Low - Security / Operations - `.env.example` contains real-looking secret values

The example env file contains values that look like usable secrets instead of placeholders.

- See:
  - [.env.example](C:/Users/song/Desktop/pcu_graduationproject_v2/.env.example#L5)

Impact:

- Even if these are not live secrets, example files with realistic credentials are easy to copy into real environments unchanged.
- This is an operational security smell.

Recommendation:

- Replace them with explicit placeholders such as `CHANGE_ME`, `example-secret`, or documented generation commands.

## Overall Assessment

The core CRUD flow, Prisma schema, API contracts, and frontend data fetching are generally aligned. The codebase is already in a reasonably coherent state, and lint/tests are green.

The main gap is not basic wiring. The main gap is policy enforcement at the backend boundary:

- CSRF protection
- publish permission enforcement
- actual enforcement of year-level upload locking

Those should be addressed before relying on this in production with real users.
