# API 서버 이전 검증 프롬프트

## 목적

이 문서는 현재 프로젝트를 다음 구조로 전환할 때, 작업을 한 번에 밀어붙이지 않고 검증 단위로 쪼개어 진행하기 위한 실행 프롬프트 모음이다.

- 프론트엔드: 기존 GitHub Pages 유지
- 백엔드 API: 새 PC 서버로 이전
- 데이터베이스: 기존 Synology NAS 유지
- 업로드 파일 저장소: 기존 Synology NAS 유지

## 전제 조건

- 새 API 서버에는 전용 도메인이 없다.
- 새 API 서버는 `https://<공인IP>` 형태로 접근한다.
- 해당 공인 IP를 SAN에 포함한 TLS 인증서는 이미 발급되어 있다.
- 브라우저는 API를 교차 출처로 호출한다.
- 이 프로젝트는 현재 프론트에서 `credentials: 'include'` 쿠키 세션을 사용한다.
- 이 프로젝트는 API 서버가 로컬 파일시스템 경로에서 업로드 파일을 직접 읽고 쓴다.

## 이행 원칙

- 한 단계에서 하나의 위험만 줄인다.
- 각 단계는 "수정", "검증", "롤백 기준"이 분명해야 한다.
- 검증이 끝나기 전에는 다음 단계로 넘어가지 않는다.
- 도메인 없는 IP 기반 API라는 제약 때문에, 인증과 쿠키 검증을 별도 게이트로 다룬다.

## 필수 주의사항

- `SameSite=None` 쿠키를 쓰려면 `Secure=true`와 HTTPS가 필요하다.
- API가 IP 주소로 동작해도 호스트 전용 쿠키는 가능하므로, 쿠키 `Domain` 속성은 굳이 설정하지 않는 편이 안전하다.
- 프론트가 `https://github-pages-domain` 이고 API가 `https://<공인IP>` 이면 CORS와 쿠키 정책을 같이 검증해야 한다.
- 현재 코드에서 `PUBLIC_BASE_URL`은 API 파일 URL 생성뿐 아니라 프론트 페이지 URL 생성에도 재사용되고 있으므로, 실제 이전 시에는 `API_PUBLIC_BASE_URL`과 `WEB_PUBLIC_BASE_URL` 분리가 필요하다.
- NAS 저장소는 원격 API로 바꾸기보다 새 서버에 마운트해서 기존 파일 경로 기반 코드를 최대한 유지하는 편이 안전하다.

## 권장 산출물

각 단계가 끝날 때마다 아래를 남긴다.

- 변경 파일 목록
- 적용한 환경 변수 목록
- 실제 검증 명령 또는 검증 절차
- 성공/실패 결과
- 롤백 방법

출력 형식:

3. 이번 이전에서 왜 수정 또는 검토가 필요한지

## 프롬프트 2. 목표 아키텍처와 네트워크 경계 확정

```md
현재 프로젝트를 아래 구조로 운영하려고 합니다.

- 프론트엔드: GitHub Pages 유지

이 조건에서 운영 가능한 목표 아키텍처를 설계해주세요.

반드시 포함할 것:
- 새 서버와 NAS 간 연결 방식 제안
- NAS 저장소를 새 서버에 마운트하는 방식 제안
- PostgreSQL 접근 경로 설계
- 외부 공개 포트와 내부 전용 포트 분리
- 최소 권한 원칙에 따른 네트워크 경계
- 장애 시 우회 또는 롤백 포인트

출력 형식:
1. 최종 아키텍처 요약
2. 네트워크 흐름
3. 저장소 마운트 전략
4. DB 연결 전략
5. 보안 주의사항
6. 운영 리스크
```

### 롤백 기준

- 아직 실제 변경 전이므로 문서 확정만 한다.

## 프롬프트 3. 환경 변수 분리안 설계

### 목표

현재 env 설계를 "웹 주소"와 "API 주소"를 구분하는 방향으로 정리한다.

- 값 예시를 IP 기반 HTTPS 환경에 맞게 제시

### 프롬프트

```md
현재 프로젝트는 PUBLIC_BASE_URL 하나에 API 파일 URL과 프론트 페이지 URL 생성 책임이 섞여 있을 가능성이 있습니다.

다음 목표에 맞는 환경 변수 분리안을 설계해주세요.
- 프론트는 GitHub Pages 유지
- API는 https://<공인IP>
- DB와 업로드 파일은 NAS 유지
- 인증은 쿠키 세션 유지

반드시 포함할 것:
- 기존 env 중 유지할 것
- 새로 추가할 env
- 이름 변경이 필요한 env
- 각 env의 예시 값
- 어떤 코드가 어떤 env를 써야 하는지

출력 형식:
1. 제안 env 목록
2. 각 env의 의미
3. 예시 값
4. 코드 수정 포인트
5. 호환성 주의사항
```

### 롤백 기준

- 새 env 설계가 기존 값과 호환되지 않으면 적용 전 보류한다.

## 프롬프트 4. NAS 파일 저장소 마운트 검증

### 목표

새 API 서버에서 NAS 파일 저장소를 기존 로컬 경로처럼 사용할 수 있는지 먼저 검증한다.

### 성공 기준

- 공개 업로드 경로 마운트 성공
- 보호 업로드 경로 마운트 성공
- API 프로세스 계정이 읽기/쓰기 가능
- 하위 디렉터리 생성 가능
- 업로드 후 다시 읽기 가능

### 프롬프트

```md
새 API 서버에서 Synology NAS의 파일 저장소를 로컬 디스크처럼 마운트해서 사용하려고 합니다.

이 프로젝트는 업로드 파일을 파일시스템 경로에 직접 저장하고, 다시 해당 경로에서 읽어 응답합니다.

다음 기준으로 검증 절차를 설계해주세요.
- 공개 파일 저장 경로
- 보호 파일 저장 경로
- 읽기/쓰기 권한
- 하위 디렉터리 자동 생성
- 대용량 파일 처리 시 주의사항
- 네트워크 끊김 또는 NAS 지연 시 장애 양상

출력 형식:
1. 사전 준비
2. 검증 절차
3. 성공 기준
4. 실패 시 원인 분류
5. 권장 마운트 옵션
6. 롤백 방법
```

### 롤백 기준

- 마운트 안정성이 부족하면 API 이전을 중단하고 NAS 상 API 유지 또는 스토리지 전략 재설계로 되돌린다.

## 프롬프트 6. IP 기반 HTTPS와 리버스 프록시 검증

### 목표

도메인 없이 `https://<공인IP>` 로 API를 서비스할 때 TLS 종단과 프록시 구성을 검증한다.

### 성공 기준

- 브라우저에서 인증서 경고 없이 접속 가능
- 인증서 SAN에 IP 포함 확인
- 프록시가 원본 IP/프로토콜 헤더를 전달
- API가 HTTPS 뒤에 있는 것으로 올바르게 인식

### 프롬프트

```md
새 API 서버는 도메인 없이 https://<공인IP> 로 서비스합니다.
이 IP에 대한 인증서는 이미 발급된 상태입니다.

이 조건에서 리버스 프록시와 TLS 구성을 검증하기 위한 작업 절차를 설계해주세요.

반드시 포함할 것:
- 인증서 체인과 SAN 점검 항목
- Nginx 또는 Caddy 기준 권장 구성 방향
- X-Forwarded-* 헤더 처리
- HTTP 차단 또는 HTTPS 강제 여부
- 브라우저 경고 발생 시 원인 분류
- API 앱이 secure cookie를 쓰기 위한 전제 조건

출력 형식:
1. 사전 확인
2. 프록시 구성 체크리스트
3. 브라우저 검증 체크리스트
4. 앱 레벨 확인사항
5. 실패 원인 분류
6. 롤백 방법
```

### 롤백 기준

- 브라우저가 인증서를 신뢰하지 않으면 다음 단계로 넘어가지 않는다.

## 프롬프트 7. CORS와 쿠키 세션 검증

### 목표

GitHub Pages에서 `https://<공인IP>` API를 호출할 때 로그인 세션이 실제로 유지되는지 검증한다.

### 성공 기준

- CORS preflight 통과
- `Access-Control-Allow-Origin` 이 정확한 Pages 주소로 반환
- `Access-Control-Allow-Credentials: true` 동작
- 로그인 후 세션 쿠키 저장
- 새로고침 후 `/api/me` 인증 상태 유지
- 로그아웃 시 쿠키 삭제

### 프롬프트

```md
프론트는 GitHub Pages, API는 https://<공인IP> 입니다.
프론트는 fetch credentials: 'include' 를 사용하고, 백엔드는 쿠키 세션 기반 인증입니다.

이 환경에서 CORS와 쿠키 세션이 실제로 동작하는지 검증하기 위한 단계별 체크리스트를 만들어주세요.

반드시 포함할 것:
- 필요한 응답 헤더
- SameSite, Secure 설정 점검
- 쿠키 Domain 속성을 비워두는 이유 검토
- 브라우저 개발자도구에서 확인할 항목
- 로그인, 인증 유지, 로그아웃 검증 절차
- 실패 시 원인별 분기

출력 형식:
1. 사전 조건
2. 서버 설정 점검
3. 브라우저 검증 절차
4. 성공 기준
5. 실패 원인 분류
6. 수정 우선순위
```

### 롤백 기준

- 로그인은 되지만 세션 유지가 안 되면 배포 전환을 보류한다.

## 프롬프트 8. Google 로그인 영향도 검증

### 목표

프론트가 GitHub Pages에 그대로 남는 상태에서 Google 로그인 흐름이 깨지지 않는지 확인한다.

### 성공 기준

- 프론트 origin이 Google 설정과 일치
- Google credential 수신 후 API 검증 성공
- API 주소가 IP여도 기존 로그인 흐름 유지

### 프롬프트

```md
이 프로젝트는 프론트에서 Google Identity Services로 credential 을 받고, API가 그 토큰을 검증하는 구조입니다.

조건:
- 프론트는 GitHub Pages 유지
- API는 https://<공인IP>
- API 전용 도메인은 없음

이때 Google 로그인 흐름이 어떤 조건에서 유지되고 어떤 조건에서 깨질 수 있는지 점검표를 만들어주세요.

반드시 포함할 것:
- Google Cloud Console에서 확인할 항목
- 프론트 origin 제약
- API IP 주소 사용이 로그인 흐름에 미치는 영향
- 실패 시 나타나는 증상
- 실제 검증 절차

출력 형식:
1. 구조 요약
2. 영향 받는 설정
3. 검증 절차
4. 실패 증상과 원인
5. 대응 방법
```

### 롤백 기준

- 운영 계정으로 로그인 재현이 안 되면 전환 중지.

## 프롬프트 9. 파일 업로드와 다운로드 검증

### 목표

새 서버 API가 NAS 저장소를 사용하면서 현재의 공개/보호 파일 흐름을 유지하는지 확인한다.

### 성공 기준

- 포스터 업로드 성공
- 이미지 업로드 성공
- 게임 파일 업로드 성공
- 공개 파일 URL 정상 표시
- 보호 파일 URL 권한 제어 정상 동작
- 파일 삭제 시 NAS 실제 파일도 정리됨

### 프롬프트

```md
이 프로젝트의 API는 업로드 파일을 NAS에 마운트된 경로로 저장하고, 이후 해당 파일을 다시 읽어서 공개 또는 보호 다운로드로 제공합니다.

다음 항목을 중심으로 검증 시나리오를 설계해주세요.
- 포스터 업로드
- 이미지 업로드
- 게임 파일 업로드
- 공개 파일 조회
- 보호 파일 다운로드 권한 제어
- 파일 삭제와 정리
- NAS 지연 또는 일시 단절 시 장애 양상

출력 형식:
1. 테스트 데이터 준비
2. 기능별 검증 절차
3. 성공 기준
4. 실패 시 관찰 포인트
5. 장애 시 복구 절차
6. 릴리즈 차단 조건
```

### 롤백 기준

- 보호 파일 권한 제어가 틀어지면 즉시 이전 배포 상태로 되돌린다.

## 프롬프트 10. URL 생성 로직 분리 수정

### 목표

현재 API 코드에서 API 주소와 웹 주소가 섞여 쓰이는 지점을 분리한다.

### 성공 기준

- API 파일 URL은 API 공개 주소 사용
- 관리자 편집 링크와 공개 페이지 링크는 웹 공개 주소 사용
- 환경 변수 이름이 역할별로 명확함

### 프롬프트

```md
현재 프로젝트에서 API 공개 주소와 웹 공개 주소가 같은 환경 변수에 묶여 있을 가능성이 있습니다.

목표:
- API 파일 URL은 https://<공인IP> 기준으로 생성
- 프론트 페이지 URL은 GitHub Pages 주소 기준으로 생성

이 목표에 맞춰 최소 수정으로 코드 변경 계획을 세워주세요.

반드시 포함할 것:
- 어떤 함수/파일을 바꿔야 하는지
- 새 env 이름 제안
- 하위 호환 전략
- 리팩터링 후 검증 항목

출력 형식:
1. 수정 대상 파일
2. 변경 이유
3. 제안 코드 구조
4. env 마이그레이션 순서
5. 검증 체크리스트
```

### 롤백 기준

- 기존 공개 URL 생성이 깨지면 즉시 env와 코드 변경을 되돌린다.

## 프롬프트 11. 배포 전환 리허설

### 목표

운영 전환 전에 스테이징 또는 제한된 운영 리허설을 통해 전체 흐름을 검증한다.

### 성공 기준

- 프론트에서 새 API로 호출 성공
- 로그인/로그아웃 정상
- 관리자 기능 정상
- 공개 프로젝트 페이지 정상
- 업로드/다운로드 정상
- 에러 로그/모니터링 포인트 확보

### 프롬프트

```md
새 API 서버 전환 전에 전체 흐름 리허설 절차를 설계해주세요.

조건:
- 프론트는 GitHub Pages
- API는 https://<공인IP>
- DB와 파일은 NAS
- 인증은 쿠키 세션

반드시 포함할 것:
- 리허설 전 준비
- 기능별 점검 순서
- 확인해야 하는 서버 로그
- 브라우저에서 확인할 항목
- 성공 판정 기준
- 실패 시 즉시 되돌릴 항목

출력 형식:
1. 준비 단계
2. 리허설 순서
3. 기능 체크리스트
4. 관측 포인트
5. Go / No-Go 기준
6. 롤백 순서
```

### 롤백 기준

- 인증, 업로드, 다운로드 중 하나라도 핵심 경로가 실패하면 No-Go.

## 프롬프트 12. 실제 전환과 롤백 런북

### 목표

운영 전환 당일 절차를 실수 없이 실행할 수 있는 런북을 만든다.

### 성공 기준

- 전환 순서가 분 단위 또는 단계 단위로 정리됨
- 담당자 없이도 따라갈 수 있는 수준
- 실패 시 롤백 순서가 명확함

### 프롬프트

```md
아래 구조로 운영 전환할 때 사용할 실제 런북을 작성해주세요.

- 프론트: GitHub Pages 유지
- API: 새 PC 서버의 https://<공인IP>
- DB: Synology NAS 유지
- 파일 저장소: Synology NAS 유지

운영 전환 런북에는 반드시 아래가 포함되어야 합니다.
- 전환 직전 체크리스트
- 적용 순서
- 각 단계의 확인 포인트
- 문제 발생 시 중단 기준
- 롤백 순서
- 전환 후 모니터링 포인트

출력 형식:
1. 전환 전 체크리스트
2. 전환 절차
3. 단계별 검증
4. 중단 기준
5. 롤백 절차
6. 전환 후 24시간 관측 포인트
```

### 롤백 기준

- 리허설에서 정한 No-Go 조건과 동일하게 적용한다.

1. 프롬프트 1로 영향 범위 동결
2. 프롬프트 2로 목표 아키텍처 확정
3. 프롬프트 3으로 env 분리안 확정
4. 프롬프트 4와 5로 NAS 연결성 검증
5. 프롬프트 6으로 IP HTTPS 검증
6. 프롬프트 7과 8로 인증 검증
7. 프롬프트 9와 10으로 코드/스토리지 흐름 검증
8. 프롬프트 11로 리허설
9. 프롬프트 12로 실제 전환

## 가장 먼저 막아야 할 리스크

- IP 기반 HTTPS는 되지만 브라우저가 인증서를 신뢰하지 않는 경우
- GitHub Pages -> API IP 간 CORS는 되지만 쿠키가 저장되지 않는 경우
- NAS 마운트는 되었지만 앱 프로세스 권한으로 쓰기가 안 되는 경우
- `PUBLIC_BASE_URL` 재사용 때문에 관리자 링크나 공개 링크가 API IP로 잘못 생성되는 경우
- NAS PostgreSQL 지연 또는 연결 불안정으로 세션/업로드 트랜잭션이 흔들리는 경우
