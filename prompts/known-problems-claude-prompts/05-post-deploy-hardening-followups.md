# Claude Prompt 05: 배포 후 하드닝 및 후순위 정리

## 목적

`known_problems.md`의 아래 세 문제는 즉시 배포 차단보다는 후속 하드닝 성격이 강하다.

- `F-08` Zod 메이저 버전 불일치
- `F-10` 로그아웃 엔드포인트 CSRF 가능성
- `F-11` 만료 세션 미정리

이 프롬프트는 Claude에게 "지금 당장 크게 뒤엎지 말고, 배포 후 안전하게 정리할 수 있는 실행 계획과 우선순위"를 뽑게 하기 위한 것이다.

## Claude에 바로 넣는 프롬프트

```md
다음 문제들은 즉시 배포 차단 이슈라기보다 배포 후 하드닝과 운영 안정성 개선 과제에 가깝다.  
성급한 대수술보다, 현실적인 우선순위와 실행 계획을 제안해줘.

대상:
1. API 와 Web 의 Zod 메이저 버전 불일치
2. 로그아웃 엔드포인트의 CSRF 가능성
3. 만료 세션 미정리로 인한 `auth_sessions` 테이블 누적

관련 파일:
- `apps/api/package.json`
- `apps/web/package.json`
- `apps/api/src/modules/auth/auth.routes.ts`
- `apps/api/src/plugins/auth.ts`
- 필요 시 세션 관련 스키마 및 운영 경로

문제 배경:
- 지금 당장 서비스가 깨질 정도는 아닐 수 있지만, 장기적으로 운영 위험과 예측 불가능성을 높인다.
- 세 문제는 중요하지만 성격이 서로 다르므로, 우선순위와 적용 타이밍을 분리해야 한다.

해야 할 일:
- 각 문제를 "지금 바로", "배포 직후", "다음 정리 사이클" 중 어디에 두는 게 맞는지 분류해라.
- 각 항목의 실제 위험과 수정 난이도를 함께 설명해라.
- 바로 고칠 수 있는 작은 조치와, 나중에 구조적으로 정리해야 할 조치를 나눠라.
- 가능하면 가장 작은 범위의 현실적인 개선안을 제안해라.

작업 원칙:
- 새 기능 개발을 막을 정도의 과한 재설계는 피하라.
- 하드닝 작업도 검증 가능하게 쪼개라.
- 보안 항목은 낮은 심각도라도 무시하지 말고, 완화책 수준까지 설명하라.
- 추정은 추정이라고 명시하라.

출력 형식:
1. 항목별 우선순위 분류
2. 각 문제의 실제 위험
3. 가장 작은 개선안
4. 나중에 더 근본적으로 정리할 수 있는 방향
5. 실행 순서 제안
6. 필요하면 후속 Claude 프롬프트 초안
```
