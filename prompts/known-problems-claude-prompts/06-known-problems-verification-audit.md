# Claude Prompt 06: Known Problems 해결 여부 검증

## 목적

이 프롬프트는 `known_problems.md` 에 적힌 문제들이 실제로 해결되었는지 검증하기 위한 것이다.  
목표는 수정이 아니라 `증거 기반 판정` 이다.

## Claude에 바로 넣는 프롬프트

```md
이 저장소에서 `known_problems.md` 에 적힌 문제들이 정말 해결되었는지 검증해줘.

중요:
- 목표는 "고쳤다고 가정"하는 것이 아니라, 각 항목이 실제로 해결되었는지 증거 기반으로 판정하는 것이다.
- 해결 여부를 추측하지 마라.
- 증거가 부족하면 `미확인` 으로 분류하라.
- 가능하면 코드, 설정, 워크플로우, 문서, 실행 결과를 함께 근거로 사용하라.
- 문제를 새로 고치는 것이 아니라 먼저 검증하라.
- 단, 검증 과정에서 명백한 미해결 문제가 보이면 그 사실을 분명히 적어라.

검증 대상:
- F-01 Prisma 마이그레이션 git 미커밋 문제
- F-02 공개 연도 API isOpen 필터 누락
- F-03 poster PATCH / asset DELETE 인가 검사 불일치
- F-04 Web ESLint 실패
- F-05 docker-compose.yml 의 하드코딩 DB 비밀번호 폴백
- F-06 CORS_ALLOWED_ORIGINS 빈 값 처리 문제
- F-07 GOOGLE_CLIENT_IDS 빈 값 처리 문제
- F-08 Zod 메이저 버전 불일치
- F-09 deploy-web-pages.yml publish_branch 불일치
- F-10 로그아웃 엔드포인트 CSRF 가능성
- F-11 만료 세션 미정리
- F-12 README 문서 불일치

검증 원칙:
1. 각 문제를 `해결됨`, `부분 해결`, `미해결`, `미확인` 중 하나로 판정해라.
2. 각 판정에는 반드시 근거를 붙여라.
3. 가능하면 관련 파일 경로와 핵심 코드/설정 위치를 적어라.
4. 실행 가능한 검증은 실제로 수행하고, 실행하지 못한 것은 왜 못했는지 적어라.
5. 문서 문제는 README 와 workflow, 실제 코드 경로가 서로 일치하는지 비교해 판정해라.
6. 보안 문제는 "형식상 수정됨" 과 "실질적으로 위험 해소됨" 을 구분해서 보라.
7. `부분 해결` 은 수정은 들어갔지만 검증이 부족하거나 우회 가능성이 남은 경우에만 써라.

작업 순서:
1. 먼저 `known_problems.md` 의 각 항목 요구사항을 체크리스트로 재정리해라.
2. 그 다음 관련 파일을 읽어서 코드, 설정, 문서를 비교해라.
3. 가능하면 아래 검증도 수행해라.
   - web lint
   - 필요한 빌드 또는 타입체크
   - migration 관련 파일 존재 여부 확인
   - workflow / README 정합성 확인
4. 마지막에 각 문제별 판정표를 작성해라.
5. 가장 마지막에는 "지금 배포 차단으로 남는 문제"만 따로 요약해라.

출력 형식:

## 1. 검증 요약
- 해결됨 개수
- 부분 해결 개수
- 미해결 개수
- 미확인 개수

## 2. 항목별 판정
각 항목마다 아래 형식을 사용:
- ID:
- 판정:
- 근거:
- 확인한 파일:
- 추가 검증 결과:
- 남은 리스크:

## 3. 실행한 검증
- 실행한 명령
- 핵심 결과
- 실패하거나 실행하지 못한 검증

## 4. 배포 관점 결론
- 지금 배포 차단으로 봐야 하는 항목
- 배포 후 보완 가능한 항목

## 5. 필요한 후속 조치
- 미해결 항목만 간단히 정리

판정 기준을 더 구체적으로 적용해라:

F-01:
- `apps/api/prisma/migrations/` 에 실제 마이그레이션이 존재하는가
- git 추적 대상인지 확인 가능한가
- schema 와 migration 흐름이 맞는가
- 신규 환경에서 `migrate deploy` 가 의미 있게 동작할 가능성이 높은가

F-02:
- `GET /api/public/years` 조회에 `isOpen = true` 조건이 실제로 들어갔는가

F-03:
- poster PATCH, asset DELETE 가 다른 편집 엔드포인트와 동일한 `DRAFT` 제약 또는 공통 권한 헬퍼를 따르는가

F-04:
- web lint 가 실제로 통과하는가
- 규칙 비활성화로 덮었는지, 구조적으로 해결했는지도 간단히 설명해라

F-05:
- `docker-compose.yml` 에 공개된 비밀번호 폴백이 제거되었는가
- `.env` 누락 시 안전하게 실패하는가

F-06:
- `CORS_ALLOWED_ORIGINS` 가 비어 있을 때 서버가 조용히 잘못 뜨지 않도록 방어가 생겼는가

F-07:
- `GOOGLE_CLIENT_IDS` 가 비어 있을 때 OAuth 검증이 모호하게 동작하지 않도록 막았는가

F-08:
- API 와 Web 의 Zod 메이저 버전이 통일되었는가
- 아니면 여전히 다르지만 이유가 있는가

F-09:
- `deploy-web-pages.yml` 의 `publish_branch` 와 문서, 실제 브랜치 전제가 일치하는가

F-10:
- 로그아웃 CSRF 문제가 실질적으로 완화되었는가
- 완화가 없다면 미해결로 분류해라

F-11:
- 만료 세션 정리 로직이 추가되었는가
- 아니면 여전히 조회 시점 정리에만 의존하는가

F-12:
- README 의 볼륨명, 배포 방식, 레포 설명, workflow 설명이 현재 코드와 일치하는가

중요:
- "코드가 있어 보인다" 수준으로 해결 판정하지 마라.
- 반드시 `문제 원인` 이 실제로 사라졌는지 보라.
- 확신이 부족하면 `미확인` 또는 `부분 해결` 로 남겨라.
```
