# Claude Prompt 04: 빌드, 배포, 문서 정합성 정리

## 목적

`known_problems.md`의 아래 세 문제를 함께 다루기 위한 Claude용 실행 프롬프트다.

- `F-04` Web ESLint 실패
- `F-09` `deploy-web-pages.yml` 의 `publish_branch` 불일치
- `F-12` README 문서 불일치

이 묶음은 "코드 자체의 핵심 비즈니스 로직보다는, 배포와 운영 정합성을 맞추는 작업"이라는 공통점이 있다.

## Claude에 바로 넣는 프롬프트

```md
다음 문제들은 코드 버그라기보다 빌드, 배포, 문서 정합성 문제다. 한 번에 정리해줘.

대상:
1. Web lint 실패 원인 정리 및 수정
2. `deploy-web-pages.yml` 의 `publish_branch` 설정과 실제 브랜치 전략 일치 여부 점검
3. `README.md` 의 현재 인프라/배포 방식 불일치 수정

관련 파일:
- `apps/web/src/app/router.tsx`
- `.github/workflows/deploy-web-pages.yml`
- `README.md`
- 필요 시 현재 배포 관련 문서 및 워크플로우 참조

문제 배경:
- lint 실패는 배포 게이트로 연결될 수 있다.
- 배포 브랜치 설정이 실제 운영 환경과 다르면 배포가 성공해도 서비스가 갱신되지 않을 수 있다.
- README 가 오래된 배포 방식과 현재 파이프라인을 섞어 설명하면 운영 실수가 발생한다.

작업 원칙:
- 불필요한 구조 변경 없이 lint 를 통과시켜라.
- 문서는 현재 실제 배포 흐름 기준으로 맞춰라.
- 확실하지 않은 외부 전제는 "확인 필요" 로 분리해라.
- 코드 정리와 문서 정리를 섞되, 범위를 불필요하게 넓히지 마라.

완료 조건:
- web lint 실패 원인이 해소된다.
- pages 배포 브랜치 설정이 실제 문서 및 운영 전제와 충돌하지 않게 정리된다.
- README 가 현재 인프라 및 배포 흐름과 어긋나지 않게 갱신된다.

출력 형식:
1. 수정 사항
2. 확인이 필요한 외부 전제
3. 검증 방법
4. 남은 운영 리스크

가능하면 수정 diff 또는 구체적인 변경안까지 제시해라.
```
