# 업로드 기능 권장 구현 방향 및 레거시 자산 대응 방안

## 목적

이 문서는 두 가지를 함께 정리한다.

1. 현재 프로젝트에서 업로드 기능을 어떻게 완성할지에 대한 권장 구현 방향
2. 기존 작품 파일을 현재 업로드 구조로 옮기지 않고 노출하는 방법, 그리고 불가피한 재배치 시 원본 삭제를 막는 방어 전략

---

## 업로드 기능 권장 구현 방향

현재 코드 기준으로 업로드 기능은 기초 인프라가 이미 상당 부분 들어와 있다.

- 멀티파트 수집
- 파일 시그니처 검사
- 이미지 WebP 변환
- 영구 저장
- DB 기록 전 실패 시 롤백

즉, 지금 필요한 일은 업로드 엔진을 새로 만드는 것이 아니라, 업로드를 비즈니스 규칙과 화면 흐름까지 완성하는 것이다.

### 1. 업로드 정책을 먼저 확정한다

가장 자연스러운 규칙은 다음과 같다.

- 포스터: `0..1개`
- 추가 이미지: `0..N개`
- 게임 ZIP: `0..1개`
- 수정 가능 상태: `DRAFT`에서만 허용

현재는 이 제약이 전부 강제되고 있지 않다.  
예를 들어 포스터 지정은 같은 프로젝트 자산인지까지만 보고, 실제로 이미지 계열 자산인지까지는 충분히 제한하지 않는다.

구현 방향:

- 포스터 지정 시 `IMAGE` 또는 `POSTER`만 허용
- `GAME`은 프로젝트당 하나만 허용
- `POSTER`도 프로젝트당 하나만 실질적으로 유지되도록 정책 결정

### 2. 서버는 파일 처리보다 업로드 규칙 강제를 먼저 마무리한다

현재 서버는 파일 자체를 저장하는 능력은 이미 있다.  
부족한 쪽은 운영 규칙이다.

우선 정리할 항목:

- 연도 업로드 허용 여부(`isUploadEnabled`) 검사
- 자산 종류별 중복 정책
- 포스터 교체 시 기존 포스터 처리 규칙
- 초안이 아닌 프로젝트 수정 차단 일관성

특히 연도 잠금은 반드시 반영해야 한다.

권장 방향:

- 제출 시 연도가 없으면 임의 생성하지 않음
- 연도가 존재하는지 먼저 확인
- `isUploadEnabled=false`이면 제출 차단

### 3. 자산 종류별 중복 정책을 코드로 명확히 강제한다

현재 구조에서는 `POSTER`와 `GAME`이 여러 개 생길 수 있다.  
운영 관점에서는 이 상태가 계속 쌓이면 관리가 매우 불편해진다.

권장 정책:

- `GAME`
  - 하나만 유지
  - 새 파일 업로드 시 기존 게임 파일을 교체하거나 업로드 자체를 거절
- `POSTER`
  - 하나만 유지
  - 새 포스터 업로드 시 이전 포스터 자산을 자동 정리하거나 비활성화

이 프로젝트는 자산 목록을 그대로 보여주므로, 포스터 교체 시 이전 포스터를 계속 남겨두는 정책은 혼란을 만들 수 있다.

### 4. 업로드 실패 복구는 현재 파이프라인을 활용한다

기존 업로드 파이프라인은 다음 흐름을 이미 갖고 있다.

- 임시 파일 추적
- 검증
- 이미지 변환
- 영구 저장
- DB 실패 시 롤백
- 최종 temp 정리

따라서 구현 순서는 다음을 유지하는 것이 좋다.

1. 업로드 전 정책 검사
2. 파일 저장
3. DB 기록
4. 후처리

후처리 예시:

- `posterAssetId` 갱신
- 기존 포스터/게임 자산 정리
- 필요 시 파일 숨김 또는 비활성화

### 5. `UploadJob`은 지금 단계에서 억지로 쓰지 않는다

현재 구조는 동기 업로드 모델이다.  
`UploadJob` 모델이 있어도 실제 플로우에 연결되어 있지 않다면, 이번 단계에서 억지로 비동기 업로드까지 확장하는 것은 범위를 불필요하게 넓힌다.

권장 판단:

- 지금은 동기 업로드를 안정화
- 대용량 파일 비동기 처리는 나중 단계에서 별도 설계

### 6. 프론트엔드는 업로드 UX를 마무리한다

새 작품 등록과 수정 화면은 기본적인 업로드 전송은 이미 한다.  
다만 사용자가 실패 원인을 미리 이해할 수 있는 UX가 약하다.

보강할 내용:

- 허용 확장자와 최대 용량 안내
- `onChange` 시 1차 파일 검증
- 업로드 중 상태 표시
- 업로드 성공/실패 메시지
- 업로드 성공 후 입력 초기화
- 중복 업로드 정책에 맞는 경고 메시지

### 7. 포스터 교체 흐름은 한 가지 패턴으로 통일한다

현재는 포스터 업로드 후 별도 `setPoster` 호출을 추가로 하는 패턴이 섞여 있다.  
이 패턴은 동작은 가능하지만 관리가 번거롭다.

권장 방향:

- `kind=POSTER` 업로드 시 서버가 자동으로 `posterAssetId`까지 갱신

이렇게 하면 프론트는 후속 호출을 따로 하지 않아도 되고, 포스터 교체 흐름이 단순해진다.

### 8. 프론트의 미리보기 메모리 정리까지 포함한다

포스터 미리보기는 `URL.createObjectURL`을 쓰고 있으므로, 이전 URL 해제 또는 cleanup 처리가 필요하다.

이 부분은 기능 핵심은 아니지만, 장시간 관리 화면 사용 시 누수 방지에 도움이 된다.

### 9. 권장 구현 순서

1. 업로드 정책 확정
2. API에서 연도 잠금, 종류 제한, 포스터 지정 가능 종류 강제
3. `POSTER`, `GAME` 교체 정책 반영
4. 프론트의 허용 형식/용량/에러 메시지/진행 상태 보강
5. 업로드 성공 후 목록 갱신과 포스터 반영 흐름 정리
6. 수동 테스트 시나리오 고정

### 10. 수동 테스트 시나리오

최소한 아래 케이스는 확인해야 한다.

1. 정상 이미지 업로드
2. 잘못된 MIME 업로드
3. 용량 초과 업로드
4. ZIP 두 개 업로드
5. 잠긴 연도 업로드
6. 초안이 아닌 프로젝트 수정 상태에서 업로드 시도
7. 포스터 삭제 시 `posterAssetId` 정리 여부

### 11. 핵심 판단

이 프로젝트는 업로드 엔진 자체는 이미 존재한다.  
남은 구현은 "파일 업로드 기능 추가"가 아니라 "업로드를 도메인 규칙에 맞게 닫는 작업"이다.

---

## 레거시 작품 자산 노출 및 안전한 마이그레이션 방안

기존 작품 파일을 현재 업로드 구조(`public/`, `protected/`)로 반드시 옮기지 않고도 노출할 수 있는 방법을 정리한다.  
불가피하게 파일을 옮겨야 한다면, 원본 삭제 사고가 절대 발생하지 않도록 방어하는 전략도 함께 제시한다.

---

## 현재 구조 요약

현재 API는 관리형 업로드 자산만 전제로 동작한다.

- 공개 자산: `/api/assets/public/:storageKey`
- 보호 자산: `/api/assets/protected/:storageKey`
- 실제 파일 경로 계산: `UPLOAD_ROOT_PUBLIC`, `UPLOAD_ROOT_PROTECTED` 아래에서 `storageKey` 기반 생성

즉, 지금 구조는 다음 가정을 깔고 있다.

1. 모든 파일은 현재 관리형 스토리지 안에 있다.
2. 모든 파일은 `storageKey`로 찾을 수 있다.
3. API가 직접 파일 경로를 계산해 읽는다.

이 구조 때문에 기존 작품 파일이 다른 위치에 있거나 외부 링크 기반이면 그대로는 노출할 수 없다.

---

## 가장 안전한 방향

가장 안전한 방향은 기존 자산과 신규 업로드 자산을 같은 방식으로 억지 통합하지 않는 것이다.

- 신규 업로드 자산: 현재 관리형 스토리지 방식 유지
- 기존 작품 자산: 레거시 자산 계층으로 별도 관리

핵심은 "파일을 옮기는 것"이 아니라 "자산을 해석하는 방법을 분기하는 것"이다.

---

## 파일을 옮기지 않고 노출하는 방법

### 1. 자산 출처를 구분한다

`Asset` 테이블 또는 별도 레거시 자산 테이블에 자산 출처 개념을 추가한다.

예시:

- `LOCAL_MANAGED`
- `LEGACY_PATH`
- `EXTERNAL_URL`
- `GOOGLE_DRIVE`

의미:

- `LOCAL_MANAGED`: 현재 업로드 기능으로 저장된 파일
- `LEGACY_PATH`: 예전 서버/NAS 디렉토리에 있는 파일
- `EXTERNAL_URL`: 외부 URL 직접 참조
- `GOOGLE_DRIVE`: 드라이브 ID 기반 다운로드 링크

이 구분이 있어야 기존 작품을 새 업로드 구조에 억지로 맞추지 않아도 된다.

### 2. 자산 해석 레이어를 만든다

현재는 `storageKey -> 실제 파일 경로`로 바로 이어진다.  
이를 `resolveAsset(asset)` 같은 해석 레이어로 바꾼다.

예시 분기:

- `LOCAL_MANAGED`
  - 현재 방식 그대로 사용
- `LEGACY_PATH`
  - 레거시 루트 경로 + DB에 저장된 상대 경로를 결합
- `EXTERNAL_URL`
  - 서버가 파일을 읽지 않고 URL 직접 반환 또는 redirect
- `GOOGLE_DRIVE`
  - 드라이브 다운로드 URL로 변환 후 redirect

이렇게 하면 자산 노출 방식이 파일 출처에 따라 달라질 수 있다.

### 3. 레거시 전용 루트를 환경변수로 분리한다

운영 환경에 다음과 같은 읽기 전용 루트를 추가한다.

- `LEGACY_PUBLIC_ROOT`
- `LEGACY_PROTECTED_ROOT`

이 경로는 기존 NAS 또는 과거 작품 디렉토리를 그대로 가리키게 한다.

중요:

- 관리형 신규 업로드 루트와 분리한다.
- 가능하면 운영체제 또는 컨테이너 마운트 수준에서 읽기 전용으로 건다.

### 4. 레거시 전용 URL 또는 API 엔드포인트를 둔다

기존 `/api/assets/public/:storageKey`에 모든 경우를 억지로 넣지 말고, 레거시 자산은 별도 경로로 분리하는 것이 안전하다.

예시:

- `/api/legacy-assets/public/:id`
- `/api/legacy-assets/protected/:id`

이 엔드포인트는 DB에서 레거시 자산 정보를 읽고, 실제 파일 경로나 외부 URL을 해석해 응답한다.

장점:

- 기존 관리형 자산 로직과 충돌이 적다.
- 삭제/권한/경로 처리 정책을 따로 둘 수 있다.
- 운영 중 문제 발생 시 영향 범위를 분리할 수 있다.

### 5. 가능한 경우 URL 직접 저장 방식을 우선한다

기존 자산이 이미 외부에서 접근 가능한 위치에 있다면 서버가 중계하지 않아도 된다.

예시:

- 포스터: 절대 URL 저장
- 게임 다운로드: 절대 URL 저장

이 경우 public API 응답에서 해당 URL을 그대로 내려주면 된다.

장점:

- 파일 이동 필요 없음
- 서버 파일 접근 로직 단순화
- 삭제 사고 위험 거의 없음

주의:

- 외부 링크의 장기 안정성을 확인해야 한다.
- 권한 제어가 필요한 파일이면 외부 URL 직노출은 부적절할 수 있다.

---

## 권장 우선순위

현실적인 적용 순서는 다음과 같다.

1. 외부 URL로 처리 가능한 자산은 URL 직접 저장
2. 서버 디스크에만 있는 기존 파일은 `LEGACY_*_ROOT` 읽기 전용 참조
3. 앞으로 올라오는 파일만 현재 관리형 업로드 구조 사용

이 방식이면 과거 데이터와 신규 데이터를 동시에 운영할 수 있고, 기존 파일을 굳이 재배치할 필요가 없다.

---

## 그래도 옮겨야 한다면

파일 재배치가 꼭 필요하다면 "이동"이 아니라 "복제 후 전환" 방식으로 해야 한다.

핵심 원칙:

- 원본 삭제 금지
- 복사 후 검증
- 검증 후 전환
- 장기간 원본 유지

---

## 절대 삭제가 일어나지 않도록 방어하는 방법

### 1. `move` 금지, `copy-only` 정책 적용

초기 마이그레이션에서는 절대 파일 이동을 하지 않는다.

- 금지: `mv`, `rename` 기반 원본 제거
- 허용: `copy`

원본은 그대로 두고, 새 위치에 복사만 한다.

### 2. 복사 매니페스트를 남긴다

복사 작업마다 아래 정보를 JSON 또는 CSV로 남긴다.

- 원본 절대경로
- 대상 절대경로
- 파일 크기
- 해시값(SHA-256)
- 복사 시각
- 검증 성공 여부
- 대응되는 DB 자산 ID

이 매니페스트는 장애 대응과 복구의 기준점이 된다.

### 3. 검증 전에는 DB를 전환하지 않는다

복사 후 다음 조건이 모두 맞아야만 서비스 참조를 새 경로로 전환한다.

1. 대상 파일 존재
2. 크기 일치
3. 해시 일치
4. 읽기 테스트 성공

하나라도 실패하면 DB는 기존 참조를 유지해야 한다.

### 4. 원본 경로는 읽기 전용으로 마운트한다

가장 강한 방어는 코드가 아니라 권한이다.

권장:

- 원본 레거시 디렉토리는 API 프로세스에 대해 읽기 전용
- 마이그레이션 도구도 원칙적으로 읽기 전용
- 복사 대상 디렉토리만 쓰기 가능

이렇게 하면 코드 버그가 있어도 원본 삭제는 발생하지 않는다.

### 5. 레거시 자산 삭제를 금지한다

현재 삭제 API는 실제 파일 `unlink`를 수행할 수 있다.  
레거시 자산에는 이 로직이 적용되면 안 된다.

방법:

- 자산에 `isManaged` 또는 `sourceType` 플래그 추가
- `LOCAL_MANAGED`만 실제 파일 삭제 허용
- 레거시 자산은 삭제 요청 시
  - 파일 삭제 금지
  - DB 상태만 `HIDDEN`, `DETACHED`, `DISABLED` 등으로 변경

즉, 레거시 자산은 "숨김"만 가능하고 "물리 삭제"는 금지한다.

### 6. 컷오버를 단계적으로 한다

권장 절차:

1. 원본 유지 + 복사만 수행
2. 샘플 작품 일부만 새 경로 참조
3. 문제 없으면 전체 전환
4. 충분한 운영 기간 뒤에도 원본은 계속 보존

중요한 점은 복사와 서비스 전환을 같은 단계에서 하지 않는 것이다.

### 7. 삭제 기능 자체를 환경변수로 잠근다

운영 보호용 플래그를 둔다.

예시:

- `ALLOW_SOURCE_DELETE=false`

정책:

- 기본값은 항상 `false`
- 이 값이 `true`가 아니면 어떤 스크립트도 원본 삭제 불가
- 초기 운영에서는 `true`로 바꾸지 않는다

이 값은 기능 플래그가 아니라 사고 방지 장치로 취급해야 한다.

### 8. 마이그레이션 스크립트는 dry-run 기본

마이그레이션 스크립트는 기본적으로 실제 복사를 하지 않고 리포트만 출력해야 한다.

리포트 항목 예시:

- 총 파일 수
- 총 용량
- 누락 파일 수
- 충돌 경로 수
- 예상 대상 경로 샘플

실제 실행은 명시적인 `--apply` 옵션이 있을 때만 허용한다.

추가 권장:

- `--delete-source` 옵션은 아예 제공하지 않는다

## 실무적으로 가장 추천하는 시나리오

이 프로젝트에서는 레거시 데이터가 외부 링크, 상대 경로, 수동 등록 방식이 섞여 있을 가능성이 높다.  
따라서 다음 방식이 가장 안전하다.

1. 레거시 자산을 신규 업로드 체계에 즉시 편입하지 않는다.
2. 레거시 자산용 출처 필드와 해석 레이어를 만든다.
3. 외부 링크는 URL 직접 저장으로 처리한다.
4. 서버 디스크 기반 레거시 파일은 읽기 전용 루트로 참조만 한다.
5. 신규 업로드만 현재 managed storage를 사용한다.

이렇게 하면:

- 기존 파일을 옮기지 않아도 된다.
- 기존 자료 훼손 위험이 매우 낮다.
- 신규 기능 개발과 과거 데이터 유지가 충돌하지 않는다.

## 결론

가장 좋은 해법은 "기존 파일을 새 구조로 옮기는 것"이 아니라 "기존 파일도 읽을 수 있게 API를 확장하는 것"이다.

요약:

- 최우선: 레거시 자산 계층 분리
- 가능하면: 외부 URL 직접 저장
- 로컬 레거시는: 읽기 전용 참조
- 부득이한 재배치는: 복사만 하고 삭제 금지

즉, 이 작업의 핵심은 스토리지 재배치가 아니라 자산 출처를 안전하게 해석하는 구조를 만드는 데 있다.
